# Native Transpilation Targets

## Overview

Ni supports multiple compilation targets from the same AST, via a **pluggable code generation backend** system. The compiler is target-agnostic -- it parses source into an AST, then hands the AST to whichever backend is requested. New backends can be added without modifying the compiler.

```
                              +-> BytecodeGen  -> .nic   (VM interpreted)
                              |
  source.ni -> Lexer -> AST --+-> RustCodeGen  -> .ni.rs (native, same ecosystem)
                              |
                              +-> CCodeGen    -> .ni.c   (portable, embed anywhere)
                              |
                              +-> ???CodeGen  -> .???    (future: Zig, LLVM IR, etc.)
```

### The CodeGen Trait

All backends implement a common trait:

```rust
pub trait NiCodeGen {
    /// Emit a complete module (file-level wrapper + all contents).
    fn emit_module(&mut self, module: &AstModule) -> Result<String>;
    /// Emit a standalone function.
    fn emit_function(&mut self, func: &AstFunction) -> Result<String>;
    /// Emit a class definition with methods.
    fn emit_class(&mut self, class: &AstClass) -> Result<String>;
    /// Emit a coroutine as a state machine.
    fn emit_coroutine(&mut self, fiber: &AstCoroutine) -> Result<String>;
    /// File extension for output (e.g., "rs", "c", "zig").
    fn file_extension(&self) -> &str;
    /// Runtime preamble (imports, includes, etc.).
    fn runtime_preamble(&self) -> &str;
    /// Human-readable name for error messages.
    fn backend_name(&self) -> &str;
}
```

The compiler drives code generation by walking the AST and calling these methods. Each backend is responsible for:

- Mapping Ni types to target language types.
- Transforming coroutines into state machines (shared utility code is available -- backends call a common `coroutine_to_state_machine(ast) -> StateMachine` transform, then emit the state machine in their target syntax).
- Emitting host API calls in the target's conventions (trait calls in Rust, function pointers in C, etc.).

### Shipped Backends

| Backend | Output | Status | Use Case |
|---------|--------|--------|----------|
| `BytecodeGen` | `.nic` | Core (always available) | User scripts, rapid iteration, modding |
| `RustCodeGen` | `.ni.rs` | Core | Engine built-ins, performance-critical paths |
| `CCodeGen` | `.ni.c` | Core | Embedding in non-Rust engines, portability |

Future backends (community or official) might target Zig, LLVM IR, JavaScript/TypeScript, or other languages -- the architecture imposes no limits.

## Rust Transpilation Target (Primary)

Because the Ni VM, compiler, and tooling are all written in Rust, Rust is the **primary native target**. Generated Rust code uses the same types and engine API crate as the VM itself -- no FFI boundary, no marshalling overhead.

### Purpose

1. **Built-in behaviors.** Common reusable behaviors can be written in Ni, transpiled to Rust, and compiled directly into the host binary. Native speed, zero interpreter overhead.

2. **Compile to Native.** Advanced users can transpile performance-critical scripts to Rust for ~10-50x speedup. Useful for data processing, algorithms, and hot paths.

3. **Type-safe integration.** Generated Rust code benefits from the Rust compiler's type checking, lifetime analysis, and optimization passes -- an extra layer of verification beyond what the Ni compiler itself provides.

### Output Conventions

The Rust transpiler emits code that:

- Uses the same `NiValue` enum as the VM (no conversion needed).
- Calls the same engine API traits directly (no FFI).
- Can interoperate with bytecode scripts seamlessly: a native-compiled function and a bytecode function use the same dispatch interface.
- Can compile to Wasm via `wasm-pack` / `wasm-bindgen`.

### Example

Ni source:

```ni
fun clamp(x: int, lo: int, hi: int) -> int:
    if x < lo:
        return lo
    if x > hi:
        return hi
    return x
```

Generated Rust:

```rust
// Auto-generated by Ni compiler. Do not edit.
use ni_runtime::prelude::*;

pub fn ni_fun_clamp(vm: &mut NiVM, args: &[NiValue]) -> NiResult<NiValue> {
    let x = args[0].as_int()?;
    let lo = args[1].as_int()?;
    let hi = args[2].as_int()?;

    if x < lo { return Ok(NiValue::Int(lo)); }
    if x > hi { return Ok(NiValue::Int(hi)); }
    Ok(NiValue::Int(x))
}
```

### Coroutine Transformation

Coroutines (`wait`, `yield`, `spawn`) are transformed to **state machines** at compile time for both native targets. In bytecode, the VM handles suspension natively by saving the fiber's stack. In Rust (and C), each yield point becomes a state variant:

```ni
fun poll_all(endpoints, interval):
    for ep in endpoints:
        check(ep)
        wait interval
```

Generated Rust (conceptual):

```rust
pub enum PollAllState {
    Check { ep_index: usize },
    WaitPause { ep_index: usize, timer: f64 },
    Finished,
}

pub fn ni_fiber_poll_all(
    vm: &mut NiVM,
    state: &mut PollAllState,
    endpoints: &[NiValue],
    interval: f64,
) -> FiberResult {
    match state {
        PollAllState::Check { ep_index } => {
            vm.call("check", &[endpoints[*ep_index].clone()])?;
            *state = PollAllState::WaitPause { ep_index: *ep_index, timer: interval };
            FiberResult::Yield
        }
        PollAllState::WaitPause { ep_index, timer } => {
            *timer -= vm.delta_time();
            if *timer <= 0.0 {
                let next = (*ep_index + 1) % endpoints.len();
                *state = PollAllState::Check { ep_index: next };
            }
            FiberResult::Yield
        }
        PollAllState::Finished => FiberResult::Done,
    }
}
```

Each `wait` / `yield` point becomes a state variant. The fiber resumes by re-entering the match at its saved state. This is the same transformation that Rust's `async/await` uses internally -- proven, well-understood, and optimizable by LLVM.

### Inheritance and Super Calls

Both backends track a `superclass_map: HashMap<String, String>` populated at the start of `emit_class_decl` (before methods are compiled). When a method body contains `super.method(args)`, the codegen resolves the parent class name from the map and emits a direct call to the parent's mangled method function:

```rust
// Rust: super.init(name) in class Dog extends Animal
Animal__init(vm, self_val, &[name])?
```

```c
// C: super.init(name) in class Dog extends Animal
Animal__init(vm, self_val, name)
```

The `superclass_map` must be populated before method compilation so that super calls inside method bodies can resolve correctly.

## C Transpilation Target (Portability)

The C target exists for embedding Ni in non-Rust host applications (C, C++, Zig, etc.) and for maximum platform portability.

### Output Conventions

The C transpiler emits code that:

- Uses a `NiValue` tagged union (NaN-boxed 64-bit values), defined in `ni_runtime.h`.
- Calls engine API functions through C function pointers (set up at initialization by the host engine).
- Uses the same coroutine-to-state-machine transformation as the Rust target.
- Compiles with any C99 compiler. No C++ features, no GCC extensions, no platform-specific code.
- Can compile to Wasm via Emscripten or `wasm32-unknown-unknown` Clang target.

### Example

Same `clamp` function, C output:

```c
// Auto-generated by Ni compiler. Do not edit.
#include "ni_runtime.h"

NiValue ni_fun_clamp(NiVM* vm, int argc, NiValue* args) {
    NiValue x  = args[0];
    NiValue lo = args[1];
    NiValue hi = args[2];

    if (ni_less_than(x, lo)) return lo;
    if (ni_greater_than(x, hi)) return hi;
    return x;
}
```

### C Runtime Header

The `ni_runtime.h` header provides:

- `NiValue` type (NaN-boxed tagged union)
- Value constructors: `ni_int(i64)`, `ni_float(f64)`, `ni_bool(bool)`, `NI_NONE`
- Type checks: `ni_is_int(v)`, `ni_is_float(v)`, `ni_is_string(v)`, etc.
- Extractors: `ni_as_int(v)`, `ni_as_float(v)`, `ni_as_string(v)`, etc.
- Operators: `ni_add(a, b)`, `ni_less_than(a, b)`, etc.
- Object operations: `ni_get_prop(obj, name)`, `ni_set_prop(obj, name, val)`, etc.
- GC registration: `NI_GC_ROOT(var)`, `ni_gc_alloc(size)`, etc.

## ABI Compatibility Across Targets

All three targets (bytecode, Rust, C) share a common calling convention through a unified dispatch layer:

```
+--------------------------------------------------+
|               Dispatch Layer                      |
|                                                   |
|  fn call(name, args) {                           |
|      match registry.get(name).target {           |
|          Bytecode(chunk) => vm.interpret(chunk)  |
|          RustNative(f)   => f(vm, args)          |
|          CNative(f)      => f(vm, args.len(),    |
|                                args.as_ptr())    |
|      }                                           |
|  }                                               |
+--------------------------------------------------+
```

- A bytecode script calling `clamp(x, 0, 100)` works identically whether `clamp` is bytecode, Rust-compiled, or C-compiled.
- The dispatch layer checks a function's `target` flag and routes accordingly.
- Mixed execution is fully supported: a bytecode script can call a Rust-compiled function which calls back into a bytecode function.

## Limitations of Native Targets

| Feature | Bytecode | Rust | C |
|---------|----------|------|---|
| Coroutines | Native (fiber stack) | State machine transform | State machine transform |
| Hot reload | Yes | No (recompile) | No (recompile) |
| GC | VM-managed | `NiValue` uses `Arc` or arena | Manual `NI_GC_ROOT` macros |
| Dynamic dispatch | Full | Trait objects / enum dispatch | Function pointer tables |
| `eval` / dynamic code | Possible (future) | Not supported | Not supported |
| Debug/breakpoints | Full source mapping | Limited (release builds) | Limited |
| Sandbox guarantees | Full | Rust type system enforced | Host engine must enforce |

**Recommendation:** Use bytecode for user-authored scripts (safety, hot-reload, debugging). Use Rust transpilation for built-in behaviors and performance-critical paths. Use C transpilation for embedding Ni in non-Rust hosts or targeting platforms without a Rust toolchain.
